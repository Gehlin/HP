# Phase 06: Audit Existing Question Bank

The general audit originally planned for "Phase 03" before the copyright fix (Phases 03–05) took priority. Compare the full `src/data/questions.ts` question bank against the `Reference/calibration/*.md` docs per subtest, and produce a gap report covering coverage, difficulty spread, and format correctness. This phase is **investigation and reporting only** — no content changes. Phase 07+ will act on whatever this finds, and some findings may need a user decision before Phase 07 is scoped (flag those explicitly rather than deciding unilaterally).

Read the root `README.md` first (copyright boundary — still applies: any gap-filling recommendations you write should describe what's needed, not draft actual new questions yet), and all 8 `Reference/calibration/*.md` docs.

## Known findings already on the table (confirm/quantify, don't re-discover from scratch)

- **ORD option count:** Phase 02 found real ORD questions use 5 options (A–E); the app's ORD questions currently use 4. Quantify exactly how many existing ORD questions this affects.
- **ELF:** no real exam content exists anywhere in the archive (UHR never publishes it) — calibration is structural facts only (10 questions/22 min, comes after MEK). Note what the app currently has for ELF and whether it's reasonable given this constraint.
- **DTK mobile width:** real DTK tables commonly run 8–10 columns; Phase 05 found the app's mobile rendering clips beyond what its existing successful DTK items use. Check whether this affects any *existing* DTK questions (not just the ones Phase 05 wrote), since this audit covers the whole bank.
- **Answer-key defects:** Phase 03 found 6 of the (now-replaced) copied items had internally-inconsistent answers, traced to `pdftotext`'s mishandling of stacked math notation during original authoring. This suggests the failure mode could exist elsewhere in the bank too — decide a reasonable sampling approach to check, since re-verifying every question in an 18,900-line file isn't practical in one phase.

- [x] Assess volume adequacy per type — not against a single real exam sitting's count (which is fixed and small, e.g. real NOG is only 6 questions per sitting), but against how much a *practice pool* needs to avoid a regular user seeing repeats too often. Use whatever session-building logic already exists (`src/utils/session.ts`'s `buildSession()` and friends) to understand how questions get selected/rotated, and give a rough recommendation per type: "fine as-is," "could use more," or "thin" — with actual numbers, not just a feeling.

  **Completion note:** Read `session.ts` end to end: `buildSession()` itself does no deduplication — `groupLinkedPassages()` only reorders ids so passage-mates stay adjacent, it never filters or excludes. The actual pool logic lives in callers; `Practice.tsx`'s plain drill flow (`start()`) filters the bank by type/difficulty/tag, **reshuffles the entire filtered pool from scratch every session**, and slices off `count` questions, with zero memory of prior sessions. Only the separate opt-in SRS "repetition" mode (`getDueQuestions`/`srs.ts`) has cross-session memory, and it's not the default flow. So volume adequacy is really about expected repeat-overlap between back-to-back sessions of the same type, not raw bank size. Using the app's own advertised single-subtest session sizes (`Practice.tsx`'s `TYPE_INFO`) and expected-overlap ≈ n²/N (session size² ÷ bank size): **NOG 6²/162=0.22 (fine as-is)**, **XYZ 12²/265=0.54, KVA 10²/183=0.55, MEK 10²/175=0.57, ORD 10²/168=0.60 (all fine for now)**, **DTK 12²/131=1.10 (thin — expect ~1 repeat per back-to-back session)**, **LAS 16²/107=2.39 and ELF 16²/100=2.56 (thin — worst of all 8, compounded by passage-grouping: only 24 LAS / 21 ELF distinct passage groups actually underlie those counts, and `groupLinkedPassages` forces whole groups to move together, so the effective draw pool is closer to 24/21 units than 107/100 items).** Recommendation: prioritize LAS/ELF volume expansion first, DTK second (compounds with its separate rendering problem below), XYZ/KVA/MEK/ORD/NOG can wait. Full table in `Reference/audit-findings.md`.
