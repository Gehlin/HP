# Phase 03: Break the App.tsx Eager Import Chain

**Scoping correction from the playbook README/Phase 01/02, based on how this actually works:** the bundle-size problem is caused specifically by whatever `questions.ts` dependency is reachable from `App.tsx`'s *eager* (non-lazy, top-level) import graph — Phase 02 traced this to `App.tsx → notifications.ts → pacing.ts → readiness.ts → questions.ts`. The ~12 remaining page-level consumers (`Bookmarks.tsx`, `Home.tsx`, `MathGuide.tsx`, `Profil.tsx`, `data/exams.ts`, etc.) are all already behind `lazy()` boundaries in `App.tsx` — their own static imports of `questions.ts` only pull it into *their own* lazy chunk (loaded when that page is visited, which needs the data anyway), not the eager entry bundle. **Converting those does not help bundle size and is out of scope** — don't do it. This phase's only goal: eliminate the one eager-reachable chain.
