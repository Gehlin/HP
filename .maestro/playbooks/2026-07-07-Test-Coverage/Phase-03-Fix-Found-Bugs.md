# Phase 03: Fix Bugs Found During Test Writing

Phase 02's testing surfaced three real bugs in `src/utils/achievements.ts` and `src/utils/readiness.ts`. This phase fixes them. Read `Phase-02-Supporting-Systems.md`'s completion notes first for the exact findings and the tests that already pin the current (buggy) behavior — those tests will need updating to assert the *fixed* behavior, not just left passing against the bug.

## Tasks

- [x] **Fix `speed_demon`:** it's declared in `ALL_ACHIEVEMENTS` but `checkAchievements()` has no code path that ever unlocks it — dead content. Read its Swedish name/description in `achievements.ts` to understand what it's supposed to reward (likely something about fast + accurate answering), then implement the actual unlock condition using data already available to `checkAchievements()` (e.g. per-question time data from `session.ts`/`loadHistory()`). Pick a genuinely achievable but meaningful threshold — don't invent an impossible one just to technically "fix" it.

  **Completion note:** Its description ("Svara snabbare än HP-standard i alla typer" — "answer faster than HP standard in all types") points straight at `analytics.ts`'s existing `timeAnalyticsByType()`, which already aggregates per-question `questionTimes` (recorded by `session.ts`'s `saveQuestionTime`) across all of `loadHistory()` and compares each of the 8 `QuestionType`s' average answer time against a real, already-defined `HP_SECONDS_PER_TYPE` table (e.g. ORD 45s, LAS/ELF 120s), producing a `ratio` (`avg/standard`, `<1` = faster). Rather than inventing a new time constant, `checkAchievements()` now imports `timeAnalyticsByType` and unlocks `speed_demon` when **all 8** of its per-type entries have `ratio !== null && ratio < 1` — i.e. the user has answered questions of every single type (not just the latest session) with an average pace faster than the HP-standard for that type. This is genuinely achievable (practice across all 8 types at a merely-brisk pace) but meaningful (a single lucky fast session in only 2–3 types does not qualify — types with no recorded time data stay `ratio: null` and correctly block the unlock, matching "i alla typer" literally). No new magic threshold was invented; the mechanism reuses `analytics.ts`'s existing HP-standard-per-type table verbatim.

- [x] **Fix `allrounder`:** currently unlocks from any 4 question types at ≥70%, without checking they share a section (verbal or quant), despite the description implying a same-section sweep. Fix the check to verify the 4 types are actually the 4 verbal types (ORD/LÄS/MEK/ELF) or the 4 quant types (XYZ/KVA/NOG/DTK) together, not an arbitrary mix.

  **Completion note:** Added two module-level constants, `VERBAL_TYPES = ['ORD','LAS','MEK','ELF']` and `QUANT_TYPES = ['XYZ','KVA','NOG','DTK']`, and an `isSingleSectionSweep` check (`VERBAL_TYPES.every(t => types.includes(t)) || QUANT_TYPES.every(t => types.includes(t))`) that now gates the unlock alongside the pre-existing `types.length === 4` and per-type `>=70%` checks. Since `types.length === 4` is still required, `isSingleSectionSweep` being true necessarily means `types` is *exactly* one of the two 4-element section sets, not a superset — so a mixed 2-verbal/2-quant session (e.g. ORD/LÄS/XYZ/KVA) no longer satisfies either `.every(...)` and correctly fails to unlock.

- [x] **Reconcile the "mastery" definition** between `achievements.ts` (21-day SRS interval) and `readiness.ts` (7-day interval). Read `srs.ts` to understand the actual interval progression the spaced-repetition algorithm produces, and decide which threshold more genuinely represents "this question is mastered" for this app's purposes — pick one number and use it consistently in both files (extract to a shared constant if that's cleaner than duplicating the literal).

  **Completion note:** `srs.ts`'s `recordAnswer()` only ever produces intervals along the fixed sequence `1 (new) → 3 → 8 → 20 → 50 → 125 → …` for a question answered correctly on every review (`newInterval = round(interval * easeFactor)`, `easeFactor` starting at 2.5). Neither of the two existing cutoffs corresponds to a value this progression ever actually produces: `readiness.ts`'s `>=7` is crossed automatically on the *second* correct review (interval jumps straight from 3 to 8, skipping 7 entirely) — so it undercounts, effectively meaning "≥6 days really" in practice; `achievements.ts`'s `>=21` sits just past the *third* review's interval of 20, so it silently demands a *fourth* review (interval=50, ~7 weeks) before ever counting — a much higher, and misleadingly-labeled, bar. Picked **`MASTERY_INTERVAL_DAYS = 20`**, extracted as a single exported constant in `srs.ts` (with a comment documenting the reasoning above) and imported by both `achievements.ts` (replacing its `>=21` check and the `mastered_10` achievement's Swedish description, now "≥20 dagar") and `readiness.ts` (replacing its `>=7` check). 20 is the exact interval value the algorithm produces after three consecutive correct reviews (~3 weeks of demonstrated retention) — a real, reachable state, not a value that quietly rounds up to the next tier, and strict enough to mean genuine long-term retention without requiring a fourth review that would make even the bronze-tier `mastered_10` achievement unreasonably slow to earn.

- [x] Update the existing tests in `achievements.test.ts` and `readiness.test.ts` that currently assert the old (buggy) behavior — they should now assert the *correct* behavior. Add new test cases specifically for `speed_demon`'s new unlock logic and `allrounder`'s same-section check.

  **Completion note:** `achievements.test.ts` went from 30 → 35 tests. The old "does NOT require the 4 types to be the same section" test (which asserted the *bug*, i.e. `.toContain('allrounder')` on a mixed ORD/LÄS/XYZ/KVA session) now asserts `.not.toContain('allrounder')` and is relabeled "(fixed bug)"; added a new dedicated full-verbal-sweep positive test (ORD/LÄS/MEK/ELF) alongside the pre-existing full-quant-sweep one, so both section shapes are covered as positive controls. The old single "speed_demon is defined but unreachable" test (which just proved the dead-content bug) was replaced with five tests built around a new `sessionWithTimes()` helper that stamps `questionTimes` onto a session: declared-in-`ALL_ACHIEVEMENTS`, no-unlock-with-partial-type-coverage (2 of 8 types timed), no-unlock-when-one-of-8-types-is-slower-than-its-own-standard, unlock-when-all-8-types-are-faster-than-standard, and a control confirming big streak/mastery/bookmarks/a perfect session alone (no `questionTimes`) still never unlocks it. The `mastered_10/50/100` describe block's title, interval literals, and `setMastered()`'s default parameter were all switched from the hardcoded `21`/`20` pair to `MASTERY_INTERVAL_DAYS`/`MASTERY_INTERVAL_DAYS - 1` (imported from `./srs`), so the boundary tests automatically track the constant instead of re-encoding it. `readiness.test.ts` went from 5 → 7 tests: its `setMastered()` helper gained an `interval` parameter (default `MASTERY_INTERVAL_DAYS`, imported from `./srs`) so the pre-existing "strong scenario" and "standard rounding" tests transparently now exercise the new shared threshold without needing behavioral changes to their own assertions (the mastery percentages they compute didn't change, since they were always about *count* of mastered questions, not the literal interval value); added two new boundary tests, one below (`MASTERY_INTERVAL_DAYS - 1` → mastery 0%) and exactly at (`MASTERY_INTERVAL_DAYS` → mastery 50%) the new threshold.
